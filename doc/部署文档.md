# 运维环境轻量化自动发布系统 部署文档

## 1. 文档目的

本文档用于说明本项目的部署方式、配置项含义、启动步骤和常见排查方法。

系统由两个进程组成：

- `ops-server`：提供 Web 页面、HTTP API、Jar 存储、发布记录管理
- `ops-agent`：连接 `ops-server`，接收发布任务，并通过 `ssh/scp` 在目标设备上执行替换

---

## 2. 目录说明

项目关键目录如下：

- `cmd/server`：Server 启动入口
- `cmd/agent`：Agent 启动入口
- `config/server.yaml`：Server 配置样例
- `config/agent.yaml`：Agent 配置样例
- `web/index.html`：前端页面
- `data/jars`：Jar 包存储目录
- `data/records.json`：发布记录文件
- `data/logs`：Agent 发布日志目录

---

## 3. 环境要求

建议环境：

- Go `1.22+`
- Linux 或 macOS
- 目标设备与 Agent 之间可通过 `ssh/scp` 连通
- 目标设备已配置 SSH 免密公钥

Agent 所在机器必须具备以下命令：

- `ssh`
- `scp`

---

## 3.1 端口开通要求

部署前请确认以下端口已按网络路径开通：

### Server 对外提供的 HTTP 端口

用途：

- 浏览器访问 Web 页面
- 页面调用 HTTP API
- Agent 下载 Jar
- Agent 建立 WebSocket 连接

端口来源：

- 由 `server.yaml` 中的 `listen_addr` 决定

默认值：

```yaml
listen_addr: ":8080"
```

因此默认需要开通：

- `8080/tcp`

如果你修改为其他端口，例如：

```yaml
listen_addr: ":18083"
```

则对应开通：

- `18083/tcp`

### 目标设备 SSH 端口

用途：

- Agent 使用 `ssh` 执行停服、启服、进程检查
- Agent 使用 `scp` 分发 Jar 到目标设备

端口来源：

- 由 `agent.yaml` 中每台设备的 `ssh_port` 决定

例如：

```yaml
devices:
  - id: "device-b"
    host: "115.190.6.202"
    ssh_user: "root"
    ssh_port: 23333
    temp_dir: "/tmp/deploy"
```

则需要从 Agent 所在机器到目标设备开通：

- `23333/tcp`

### 推荐的开通矩阵

- 浏览器/用户终端 -> Server：放行 `listen_addr` 对应端口
- Agent -> Server：放行 `listen_addr` 对应端口
- Agent -> 目标设备：放行每台设备的 `ssh_port`

### 典型示例

如果部署参数如下：

- Server：`listen_addr=:8080`
- 目标设备 B：`ssh_port=23333`

则至少需要开通：

- 浏览器 到 Server：`8080/tcp`
- Agent 到 Server：`8080/tcp`
- Agent 到 115.190.6.202：`23333/tcp`

### 验证方式

验证 Server 端口：

```bash
curl http://<server-host>:<port>/api/agent/status
```

验证目标设备 SSH 端口：

```bash
ssh -p <ssh_port> <ssh_user>@<device_host> 'echo ok'
```

## 4. 构建方式

在项目根目录执行：

```bash
go build -o bin/ops-server ./cmd/server
go build -o bin/ops-agent ./cmd/agent
```

也可以直接运行：

```bash
go run ./cmd/server
go run ./cmd/agent
```

## 5. Server 部署

### 5.1 配置文件

参考 [server.yaml](/Users/lvyong/Documents/code/ops/config/server.yaml)：

```yaml
listen_addr: ":8080"
public_base_url: "http://127.0.0.1:8080"
jar_dir: "data/jars"
record_file: "data/records.json"
ws_path: "/ws/agent"
```

字段说明：

- `listen_addr`：Server 监听地址
- `public_base_url`：Agent 下载 Jar 时使用的外部访问地址
- `jar_dir`：Jar 包存储目录
- `record_file`：发布记录文件，NDJSON 格式
- `ws_path`：Agent WebSocket 接入路径

### 5.2 启动命令

```bash
./bin/ops-server --config config/server.yaml
```

默认入口：

- Web 页面：`http://<server-host>:<port>/`
- Agent 状态：`GET /api/agent/status`
- 历史记录：`GET /api/tasks`

## 6. Agent 部署

### 6.1 配置文件

参考 [agent.yaml](/Users/lvyong/Documents/code/ops/config/agent.yaml)：

```yaml
server:
  ws_url: "ws://127.0.0.1:8080/ws/agent"
  jar_base_url: "http://127.0.0.1:8080/api/jars/download"
agent:
  id: "agent-a"
  workspace: "data/agent-workspace"
  log_dir: "data/logs"
devices:
  - id: "device-b"
    host: "10.0.0.2"
    ssh_user: "deploy"
    ssh_port: 22
    temp_dir: "/tmp/deploy"
services:
  - device_id: "device-b"
    service_name: "order-service"
    deploy_dir: "/srv/order-service"
    target_jar_name: "app.jar"
    start_script: "/srv/order-service/bin/start.sh"
    stop_script: "/srv/order-service/bin/stop.sh"
    process_name: "order-service"
```

字段说明：

`server`

- `ws_url`：Agent 连接 Server 的 WebSocket 地址
- `jar_base_url`：Jar 下载地址前缀

`agent`

- `id`：Agent 标识
- `workspace`：Jar 下载和临时文件目录
- `log_dir`：发布日志目录

`devices`

- `id`：设备标识，供页面和任务使用
- `host`：目标设备 IP 或域名
- `ssh_user`：SSH 用户
- `ssh_port`：SSH 端口
- `temp_dir`：Jar 上传到远端后的临时目录

`services`

- `device_id`：服务归属的设备 ID
- `service_name`：页面中使用的服务名
- `deploy_dir`：远端部署目录
- `target_jar_name`：替换后的目标 Jar 文件名
- `start_script`：启动脚本
- `stop_script`：停止脚本
- `process_name`：用于 `pgrep -f` 检测进程的匹配串

### 6.2 启动命令

```bash
./bin/ops-agent --config config/agent.yaml
```

启动后，Server 页面应显示 Agent 在线及设备列表。

## 7. 目标设备要求

每台被管理设备至少需要满足：

- Agent 所在机器可以免密 SSH 登录
- 远端存在可写的 `temp_dir`
- 远端服务具备明确的停服脚本和启服脚本
- `process_name` 可以被 `pgrep -f` 稳定匹配

推荐远端目录结构：

```text
/srv/order-service/
├── app.jar
└── bin/
    ├── start.sh
    └── stop.sh
```

发布流程为：

1. Agent 从 Server 下载 Jar
2. Agent 用 `scp` 将 Jar 上传到目标设备 `temp_dir`
3. Agent 执行 `stop_script`
4. Agent 轮询 `pgrep -f <process_name>`，等待进程退出
5. Agent 将远端临时 Jar 移动到 `<deploy_dir>/<target_jar_name>`
6. Agent 执行 `start_script`
7. Agent 再次轮询 `pgrep -f <process_name>`，确认进程启动

## 8. 启动顺序

建议启动顺序：

1. 启动 `ops-server`
2. 确认 `http://<server>/api/agent/status` 可访问
3. 启动 `ops-agent`
4. 打开 Web 页面，确认显示 Agent 在线

## 9. 验证部署成功

### 9.1 Server 验证

```bash
curl http://127.0.0.1:8080/api/agent/status
curl http://127.0.0.1:8080/api/tasks
```

### 9.2 Agent 验证

页面顶部显示：

- `Agent 在线`
- 设备列表正常展示

### 9.3 端到端验证

1. 上传一个非空 `.jar`
2. 选择服务和目标设备
3. 点击“开始发布”
4. 观察任务是否进入终态
5. 登录远端设备，检查：

```bash
ls -l <deploy_dir>/<target_jar_name>
pgrep -af '<process_name>'
```

## 10. 常见问题

### 10.1 页面显示 Agent 离线

排查顺序：

- `ops-agent` 进程是否启动
- `agent.yaml` 中 `ws_url` 是否正确
- Server 的 `ws_path` 是否与 Agent 配置一致
- 网络和防火墙是否允许访问

### 10.2 上传成功但发布失败

排查顺序：

- 上传文件是否为非空 `.jar`
- `service_name` 与 Jar 是否匹配
- 目标设备是否勾选
- `ops-agent` 日志中是否出现下载、scp、stop、start 失败信息

### 10.3 设备启动超时

通常原因：

- `start_script` 实际没有拉起服务
- `process_name` 配置不正确，`pgrep -f` 匹配不到真实进程
- 启动时间超过当前等待窗口

### 10.4 设备停止超时

通常原因：

- `stop_script` 没有真正停掉进程
- `process_name` 过宽，匹配到了不该匹配的进程

---

## 11. 生产建议

- 将 `ops-server` 和 `ops-agent` 配置为系统服务
- 为 `jar_dir`、`record_file`、`log_dir` 配置独立持久化目录
- `public_base_url` 使用 Agent 可访问的真实地址，避免使用 `127.0.0.1`
- 对目标设备统一 SSH 公钥和脚本规范
- 对 `process_name` 做单独验收，确保 `pgrep -f` 稳定可靠
