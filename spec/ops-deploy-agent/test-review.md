## 审查总结

`test.md` 已经具备较完整的测试框架，覆盖了主要单元测试和 5 个核心集成场景，但当前还没有达到“可直接执行并稳定回填结果”的质量。主要问题集中在文档状态陈旧、关键恢复链路覆盖不足，以及部分测试期望仍保留未定口径。

## P0 - 必须修复（阻塞性问题）

未发现。

## P1 - 建议修复（重要但不阻塞）

### [P1-1] 文档头部仍声明“尚未进入实现阶段”，与当前 spec/plan 状态冲突
- **类型**：文档不一致
- **位置**：`spec/ops-deploy-agent/test.md:6`
- **描述**：测试文档头部写明“当前项目尚未进入实现阶段”，但当前 `plan.md` 已为“已确认”且 TODO 已勾选完成，项目也已经进入实现与验证阶段。这个表述会误导后续执行者把文档当成纯测试设计稿，而不是可实际回填执行结果的测试文档。
- **影响**：测试执行责任和阶段判断会被混淆，可能导致结果迟迟不回填，或忽略已实现系统的真实验证。
- **建议**：把说明改为“当前项目已进入实现/验证阶段，本文档用于执行测试并回填结果”，或拆成“测试规格 + 执行记录”两个部分。

### [P1-2] Server 重启恢复链路的关键行为没有被完整测试
- **类型**：测试覆盖缺失
- **位置**：`spec/ops-deploy-agent/plan.md:170-180`，`spec/ops-deploy-agent/test.md:91-97`，`spec/ops-deploy-agent/test.md:220-236`
- **描述**：Plan 明确把“任务创建即持久化”“Server 启动时加载执行中记录到内存”“Agent 重连后继续上报结果”作为关键恢复策略。但当前测试文档只有 `S5-7` 覆盖“GetTask 能查历史记录”，集成场景 4 只验证“Agent 重新在线”，没有验证“Server 重启前创建的执行中任务，在重启后仍能接收同一 task_id 的 TaskReport 并正确更新 records.json”。
- **影响**：最关键的重启恢复设计没有被测试闭环保护，后续即使 Agent 能重连，也可能出现任务状态卡在“执行中”却无人发现。
- **建议**：新增至少 1 个单元测试和 1 个集成场景：
  - 单元：Manager 在无内存 Task 的情况下处理 `TaskReport`，仍能正确更新记录。
  - 集成：发布进行中重启 Server，待 Agent 完成后验证该 task_id 最终状态从“执行中”更新为终态。

### [P1-3] 重复上传同名 Jar 的测试口径仍是开放项，但 plan 已不允许继续模糊
- **类型**：测试期望不明确
- **位置**：`spec/ops-deploy-agent/test.md:53`，`spec/ops-deploy-agent/test.md:277`
- **描述**：`S2-6` 的期望结果写成“仅保留最新一条（或两条均保留，取决于实现约定，需在实现时确认）”，同时“遗留问题”又写“未在 plan 中明确”。这与当前文档状态冲突。测试文档的职责是给出确定的验收口径，不应把同一条用例保留成多种答案。
- **影响**：执行测试时无法客观判定通过/失败；不同执行人会得出不同结论。
- **建议**：把 `S2-6` 改成单一、可断言的期望；如果 plan 仍不足以支撑该决策，就应先回补到 plan，再同步收敛测试文档。

## P2 - 可选优化（锦上添花）

### [P2-1] 缺少对配置加载 TODO 的显式测试条目
- **类型**：文档补充
- **位置**：`spec/ops-deploy-agent/plan.md:126-130`，`spec/ops-deploy-agent/plan.md:211-222`，`spec/ops-deploy-agent/test.md:26-140`
- **描述**：Plan 中 `TODO-S1` 和 `TODO-A1` 都有明确验收标准，但测试文档没有给出对应的单元测试或轻量集成验证条目。
- **影响**：配置缺失、默认值、错误提示这些启动期问题容易被遗漏。
- **建议**：补 2 组基础测试：
  - Server 配置文件缺失/字段缺失时返回明确错误。
  - Agent 配置文件缺失/字段缺失时返回明确错误。

### [P2-2] HTTP API / 前端测试被统称为“后续覆盖”，但可以更细化
