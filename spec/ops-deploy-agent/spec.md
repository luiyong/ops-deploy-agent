# 运维环境轻量化自动发布系统 需求规格说明

> 状态：已确认
> 创建日期：2026-04-05
> 所属里程碑：无

---

## 1. 概述

### 1.1 功能简述

构建一套适用于受限内网测试环境的轻量化微服务自动发布系统：运维人员通过 Server 端 Web 页面触发发布任务，Server 将指令下发至运行在内网设备 A 上的 Agent，Agent 完成 Jar 包下载、分发与远程替换，最终将结果汇报至 Server 供查询。

### 1.2 所属模块

本系统为独立建设项目，包含两个子系统：
- **控制端（Server）**：独立服务器，承担任务管理、Jar 包存储、指令下发、发布记录展示
- **发布 Agent（运行于设备 A）**：内网发布执行节点，承担 Jar 下载、SCP 分发、SSH 远程替换

### 1.3 关联文档

- `doc/运维环境AGENT需求梳理.md`

> **架构说明**：原始需求中描述"设备 A 从外部指定位置下载 Jar 包"，经需求澄清后确认调整为"运维人员将 Jar 包上传至 Server，Agent 从 Server 以 HTTP 方式下载"。原因：设备 A 本身也是内网机器，统一由 Server 托管制品更便于管理，且运维人员无需直接操作内网设备。此调整已在制品来源、输入项、核心行为等章节统一体现。

## 2. 用户故事与场景

### 2.1 目标用户

负责测试环境发布的运维或开发人员，无法直接 SSH 登录内网设备，需借助跳板机访问。

### 2.2 用户故事

- 作为运维人员，我希望在 Server Web 页面上传 Jar 包并触发发布，以便无需手动登录内网设备逐台操作。
- 作为运维人员，我希望在 Web 页面查看每次发布的状态和结果，以便快速定位失败原因。
- 作为运维人员，我希望发布失败时能看到明确的失败节点和原因，以便手动介入处理。

### 2.3 使用场景

**场景一：正常发布**
- 前置条件：Agent 已在设备 A 上运行并与 Server 保持连接；目标设备 B/C 的启停脚本正常可用；Jar 包已上传至 Server。
- 操作：运维人员在 Web 页面选择服务名称、Jar 包、目标设备，点击"发布"。
- 期望结果：B/C 的服务依次停止、Jar 包被替换、服务以新 Jar 包重新启动；Web 页面展示各设备发布成功。

**场景二：某台设备替换失败**
- 前置条件：同场景一，但设备 B 的启动脚本异常。
- 操作：触发发布任务后，B 替换失败，C 正常执行。
- 期望结果：Web 页面展示 B 失败（含失败原因）、C 成功；运维人员可据此手动处理 B。

**场景三：Agent 离线**
- 前置条件：设备 A 上的 Agent 未运行或与 Server 断开连接。
- 操作：运维人员在 Web 页面触发发布。
- 期望结果：Server 显示 Agent 离线，发布任务无法下发，页面给出明确提示。

## 3. 功能需求

### 3.1 输入

| 输入项 | 来源 | 格式 | 是否必填 |
|--------|------|------|----------|
| 服务名称 | Web 页面 / 配置文件 | 字符串 | 必填 |
| Jar 包文件 | 用户上传至 Server | `.jar` 文件 | 必填 |
| 目标设备列表 | Web 页面选择 / 配置文件 | 设备标识列表 | 必填 |
| 远程部署目录 | 配置文件（每台设备单独配置） | 路径字符串 | 必填 |
| 启动脚本路径 | 配置文件（每台设备单独配置） | 路径字符串 | 必填 |
| 停止脚本路径 | 配置文件（每台设备单独配置） | 路径字符串 | 必填 |

### 3.2 输出

| 输出项 | 目标 | 格式 |
|--------|------|------|
| 各设备发布状态 | Server Web 页面 | 成功 / 失败 + 失败原因 |
| 发布记录 | Server 持久化存储 | 结构化日志（含时间、服务、Jar、设备、结果） |
| Agent 执行日志（含远程替换的 SSH 输出） | 设备 A 本地文件 | 文本日志（集中记录，每次发布一个日志文件） |

### 3.3 核心行为描述

#### 3.3.1 Server 端

**Jar 包管理**
- 运维人员可通过 Web 页面上传 Jar 包至 Server。
- Server 保存 Jar 包并在页面展示可用包列表（服务名 + 文件名）。

**发布任务触发**
- 运维人员在 Web 页面选择服务名称、目标 Jar 包、目标设备，点击"发布"按钮。
- Server 创建发布任务，记录初始状态为"待下发"。
- Server 检查对应 Agent 是否在线：
  - 若 Agent 在线，通过持久连接将发布指令下发给 Agent，任务状态更新为"执行中"。
  - 若 Agent 离线，任务状态更新为"失败"，页面展示"Agent 离线，无法下发"。

**结果接收与展示**
- Server 接收 Agent 上报的各设备执行结果，更新任务状态。
- Web 页面展示：任务整体状态、各设备状态、失败原因、发布时间。

**发布记录**
- 每次发布任务（无论成功与否）均持久化保存，记录至少包含：发布时间、服务名称、Jar 包名称、目标设备、各设备结果（成功/失败/失败原因）。
- 运维人员可在 Web 页面查询历史发布记录。

#### 3.3.2 Agent 端（运行于设备 A）

**Agent 启动与连接维持**
- Agent 启动后，主动与 Server 建立 WebSocket 持久连接。
- Agent 持续维持该连接，断线后自动重连。
- Server 页面展示 Agent 的在线/离线状态。

**接收指令**
- Agent 通过 WebSocket 连接接收 Server 下发的发布指令，指令包含：服务名、Jar 包下载地址（Server 提供）、目标设备列表、各设备的部署目录与启停脚本路径。

**Jar 包下载**
- Agent 从 Server 指定地址以 HTTP 方式下载 Jar 包（暂不需要鉴权）。
- 下载完成后对文件进行基本校验（文件大小不为零、文件可读）。
- 下载的 Jar 包保存在设备 A 本地指定目录中。
- 下载失败则终止任务，将失败原因上报 Server。

**Jar 包分发（串行）**
- Agent 依次对每台目标设备（B、C）通过 SCP 将 Jar 包传输到该设备的临时目录。
- 某台设备分发失败时，记录该设备失败原因，继续执行下一台。

**远程替换（串行）**
- Jar 包分发完成后，Agent 依次通过 SSH 对每台目标设备触发替换流程：
  1. 执行停止脚本，停止目标服务。
  2. 等待进程退出，最长等待 1 分钟；超时仍未退出则判定停止失败，终止该设备的替换流程。校验进程已不存在（通过进程名或 PID 检查）。
  3. 将新 Jar 包从临时目录移至服务运行目录（覆盖旧 Jar 包）。
  4. 执行启动脚本，拉起服务。
  5. 等待服务启动，最长等待 1 分钟；超时仍未检测到进程则判定启动失败。检查服务进程是否存在，以确认服务已使用新 Jar 包启动。
  6. 记录该设备替换结果（成功/失败 + 原因）。
- 某台设备替换失败，不影响后续设备的替换执行。

**结果上报**
- 所有设备执行完成后，Agent 将汇总结果上报给 Server。
- Agent 将整个发布过程（含各设备的 SSH 远程执行输出）集中写入设备 A 本地日志文件，每次发布生成一个独立日志文件，供人工排查使用。
- 设备 B/C 不要求本地写入替换日志，相关执行输出由 Agent 在 A 侧统一捕获记录。

#### 3.3.3 设备 B/C 端

- 设备 B/C 上预置有服务的启动脚本和停止脚本（本次不负责创建，需已存在）。
- 所有替换操作均由 Agent 通过 SSH 远程触发，设备 B/C 不需要主动参与。
- 设备 B/C 不要求本地写入替换日志；替换过程的 SSH 执行输出由 Agent 在设备 A 侧统一捕获并记录。

### 3.4 状态定义

**任务级状态**（Server 维护，描述整体发布任务进度）：

| 状态 | 含义 | 触发时机 |
|------|------|---------|
| 待下发 | 任务已创建，等待下发给 Agent | 用户点击发布后立即进入 |
| 执行中 | Agent 已接收指令，正在执行 | Agent 确认收到指令后 |
| 成功 | 所有目标设备均替换成功 | Agent 上报全部设备成功 |
| 部分失败 | 至少一台设备替换失败，其余成功 | Agent 上报含失败设备的结果 |
| 失败 | 任务未能执行（Agent 离线）或全部设备失败 | Agent 离线 / 全部设备均失败 |

**设备级状态**（每台目标设备独立维护）：

| 状态 | 含义 |
|------|------|
| 等待中 | 排队等待执行（串行场景中） |
| 执行中 | 正在进行替换流程 |
| 成功 | 替换完成，新进程已拉起 |
| 失败 | 替换过程某步骤失败，含失败原因 |

### 3.5 业务规则与约束

1. 同一服务的发布任务必须串行执行，不支持并发发布同一服务。
2. 发布任务中多台目标设备的替换串行执行（优先保证可靠性）。
3. 单台设备替换失败不中断其他设备的替换，但整体任务标记为部分失败。
4. Agent 对 Server 的连接断线后应自动重连，不需人工干预。
5. 服务启停脚本由目标机器已有脚本承担，Agent 不创建或修改启停脚本。
6. 暂不支持自动回滚，替换失败后由人工介入处理。

## 4. 非功能需求

### 4.1 性能要求

无特殊要求。测试环境场景，不要求高并发发布能力。

### 4.2 安全要求

- Server 与 Agent 之间的通信暂不要求加密鉴权（内网环境，后续可迭代）。
- Jar 包下载暂不要求鉴权。
- SSH 认证方式沿用现有设备 A 到 B/C 的 SSH 免密配置，Agent 不额外管理 SSH 密钥。
- Web 页面暂不要求登录鉴权（后续可迭代）。

### 4.3 兼容性要求

- 服务启停脚本需与目标设备现有脚本兼容，不依赖 systemd/systemctl 等服务管理框架。
- Agent 应能在常见 Linux 发行版上运行。

## 5. 边界与限制

### 5.1 明确包含（In Scope）

- Server 端：Jar 包上传与存储、Web 页面触发发布、发布历史记录查询、Agent 在线状态展示
- Agent 端：与 Server 建立持久连接并接收指令、Jar 包下载、SCP 分发、SSH 远程替换、结果上报
- 替换流程：停止服务、进程校验、Jar 替换、启动服务、进程校验
- 发布记录：每次发布的时间、服务、Jar、设备、结果的持久化记录

### 5.2 明确排除（Out of Scope）

- 自动回滚机制
- 蓝绿发布 / 灰度发布 / 无停机发布
- 多环境统一发布平台
- 权限审批流 / Web 登录鉴权
- 制品仓库治理（Nexus/Artifactory 集成）
- JDK 版本管理 / 第三方依赖联动升级
- 容器化 / Kubernetes 支持
- Agent 到 B/C 的 SSH 密钥自动配置（依赖现有免密配置）
- 服务启停脚本的创建或修改（依赖目标机器已有脚本）

### 5.3 已知约束

- 设备 A 为内网机器，Server 无法主动连接 A，需由 Agent 主动向 Server 建立并维持持久连接。
- 设备 B/C 不可访问外部网络，Jar 包只能由 A 通过 SCP 分发。
- 目标设备已有启停脚本的稳定性不在本系统保障范围内。
- 设备 A 到 B/C 的 SSH 免密配置需提前保障，不在本系统建设范围内。

## 6. 验收标准（Acceptance Criteria）

**AC-1：Agent 上线**
- **Given** Agent 在设备 A 上启动
- **When** Agent 运行后
- **Then** Server Web 页面显示 Agent 在线状态

**AC-2：正常发布成功**
- **Given** Agent 在线，Jar 包已上传 Server，B/C 的启停脚本可用
- **When** 运维人员在 Web 页面选择服务、Jar 包、目标设备 B 和 C，点击发布
- **Then** B 和 C 依次完成停服、Jar 替换、重启；Web 页面展示两台设备均发布成功；发布记录中有对应条目

**AC-3：单台设备失败不影响其他设备**
- **Given** Agent 在线，设备 B 的启动脚本异常
- **When** 触发对 B、C 的发布任务
- **Then** B 替换失败（含失败原因），C 正常替换成功；整体任务标记为部分失败

**AC-4：Agent 离线时无法发布**
- **Given** Agent 未运行
- **When** 运维人员触发发布任务
- **Then** Web 页面提示 Agent 离线，任务不执行

**AC-5：发布记录可查**
- **Given** 已完成若干次发布（含成功和失败）
- **When** 运维人员在 Web 页面查看发布记录
- **Then** 可查到每次发布的时间、服务名、Jar 包名、目标设备、各设备结果及失败原因

**AC-6：服务以新 Jar 包启动**
- **Given** 替换流程完成
- **When** 检查目标设备
- **Then** 运行目录中的 Jar 文件为本次发布的新版本，且替换操作完成后检测到新进程已拉起（注：不做进程实际加载 Jar 的深层校验，可后续迭代）

**AC-7：Agent 断线自动重连**
- **Given** Agent 与 Server 连接意外断开
- **When** 等待片刻
- **Then** Agent 自动重新连接 Server，无需人工干预
